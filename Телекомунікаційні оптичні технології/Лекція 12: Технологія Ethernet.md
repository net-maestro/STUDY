## 1. Стандарти для мідних кабелів (100Base-T4)

### 🔹 Особливості 100Base-T4

```
Призначення: робота на існуючій інфраструктурі (UTP Cat.3/4)
Кабель: 4 пари UTP Cat.3/4/5 (всі 8 контактів RJ-45 задіяні)
Кодування: 8B/6T + NRZI
Дуплекс: ТІЛЬКИ напівдуплекс (не підтримує full-duplex)
```

### 🔹 Розподіл пар у 100Base-T4

| Пара | Призначення | Напрямок |
|------|-------------|----------|
| Пара 1 | Передача даних | Односпрямована |
| Пара 2 | Прийом даних | Односпрямована |
| Пара 3 | Передача/Прийом | Двонаправлена ⚡ |
| Пара 4 | Передача/Прийом + виявлення колізій | Двонаправлена ⚡ |

> ⚠️ **Важливо**: 3 пари працюють одночасно на передачу даних, 4-та — для детектування колізій.

### 🔹 Обмеження довжини кабелю

```
Проблема:
• Швидкість зросла в 10× (10 → 100 Мбіт/с)
• Механізм CSMA/CD залишився незмінним
• Час передачі мінімального кадру (512 біт) скоротився

Наслідок:
• Максимальний діаметр мережі зменшено до ~205-210 м
• Для сегмента: 100 м (90 м постійна проводка + 10 м патч-корди)

📐 Правило планування:
Загальна довжина = кабель у стіні + патч-корд ПК + патч-корд комутатора ≤ 100 м
```

## 2. Концентратори в Fast Ethernet та обмеження топології

### 🔹 Типи концентраторів (хубів)

| Клас | Особливості | Макс. кількість між вузлами | Затримка |
|------|-------------|---------------------------|----------|
| **Клас I** | Конвертує сигнали між різними стандартами (TX/FX/T4) → цифровий формат → назад | **1** | Велика (через конверсію) |
| **Клас II** | Працює тільки з однаковим типом середовища, без конверсії | **2** | Мінімальна (пряма передача) |

### 🔹 Таблиця обмежень довжини (багатосегментні конфігурації)

| Конфігурація | 1× Клас I | 1× Клас II | 2× Клас II |
|--------------|-----------|------------|------------|
| **Мідь (TX/T4)** | 205 м | 200 м | 200 м |
| **Оптика (FX)** | 228 м | 272 м | 320 м |
| **Мікс: TX + FX** | 216.2 м | 260.8 м | 308.8 м |
| **Мікс: T4 + FX** | — | 231 м | — |

### 🔹 Подолання обмежень топології

```
❌ Проблема: діаметр мережі ≤ 205 м (для міді + хаб Класу I)

✅ Рішення 1: Використання комутаторів (switches)
• Кожен порт комутатора — окремий домен колізій
• Діаметр зростає: 200 м → 400 м (з одним комутатором)

✅ Рішення 2: Повнодуплексна оптика між комутаторами
• 100BASE-FX full-duplex: до 2 км (багатомод) / до 10+ км (одномод)
• Ідеально для магістралей (backbone)

🔄 Принцип:
Комутатор розбиває єдиний домен колізій на кілька незалежних сегментів
→ менше колізій → вища продуктивність
```

## 3. Взаємодія вузлів мережі

### 🔹 Базові параметри кадру

| Параметр | Значення | Примітка |
|----------|----------|----------|
| **Мінімальний кадр** | 64 октети (512 біт) | Гарантує виявлення колізій |
| **Максимальний кадр** | 1518 октетів (12144 біт) | Включає заголовки + FCS |
| **Поле даних (Payload)** | 46–1500 байт | Менше 46 → доповнення (pad) |

### 🔹 Формат MAC-адреси (48 біт) — детальний розбір

```
Біти:  47 46 ... 2 | 1 | 0
       │           │   │
       │           │   └─ I/G (Individual/Group)
       │           │      0 = Unicast (індивідуальна)
       │           │      1 = Multicast/Broadcast (групова)
       │           │
       │           └─ U/L (Universal/Local)
       │              0 = Глобальна (заводська, OUI)
       │              1 = Локальна (задана адміністратором)
       │
       └─ OUI (24 біти) + OUA (22 біти)
          • OUI: Organizationally Unique Identifier (виробник)
          • OUA: Organizationally Unique Address (пристрій)
```

### 🔹 Поле «Довжина/Тип» (Length/Type)

```
Значення поля (2 байти):

┌─────────────────┬────────────────────────┐
│ 0–1500 (0x0000–0x05DC) │ = ДОВЖИНА даних │
│                        │ (кадр 802.3)    │
├─────────────────┼────────────────────────┤
│ ≥1536 (0x0600+)  │ = ТИП протоколу      │
│                        │ (кадр Ethernet II)│
└─────────────────┴────────────────────────┘

📌 Приклади EtherType:
• 0x0800 — IPv4
• 0x0806 — ARP
• 0x86DD — IPv6
• 0x8100 — VLAN-tagged (802.1Q)
```

### 🔹 Контрольна сума (FCS — Frame Check Sequence)

```
• Алгоритм: CRC-32 (32-бітне значення)
• Обчислюється: від першого байта DA до останнього байта даних
• Перевірка: на приймальній стороні
• Якщо CRC не збігається → кадр відкидається без сповіщення
• Відновлення: забезпечується протоколами вищих рівнів (TCP)
```

## 4. Об'єднання сегментів: мости, комутатори, маршрутизатори

### 🔹 Порівняння пристроїв 2-го та 3-го рівнів

| Пристрій | Рівень OSI | Критерій пересилання | Домен колізій | Домен широкомовлення |
|----------|-----------|---------------------|---------------|---------------------|
| **Концентратор (Hub)** | Фізичний (1) | Не аналізує, повторює на всі порти | Один на всі порти | Один |
| **Міст / Комутатор** | Канальний (2) | MAC-адреса призначення | Окремий на кожен порт | Один (без VLAN) |
| **Маршрутизатор** | Мережевий (3) | IP-адреса / префікс мережі | Окремий на кожен інтерфейс | Розділяє |

### 🔹 Логічна структуризація мережі

```
🎯 Мета: локалізація трафіку
• Трафік, призначений для сегменту, не виходить за його межі
• Зменшення навантаження на кожен сегмент
• Зниження ймовірності колізій (для CSMA/CD)
• Підвищення безпеки та керованості

🔧 Інструменти:
• Мости (застарілі, програмні)
• Комутатори (сучасні, апаратні, high-performance)
• Маршрутизатори (для міжсегментної комунікації)
```

> 💡 **Принцип**: "Switch where you can, route where you must"

## 5. Структура та робота комутаторів

### 🔹 Три основні архітектури комутаторів

#### 1️⃣ Комутаційна матриця (Switching Matrix)

```
[Порт 1] ──┐
[Порт 2] ──┼── [Матриця 3-рівнева] ──┬── [Порт N]
[Порт 3] ──┘                         └── [Порт M]

• Найшвидший спосіб (апаратна комутація)
• Перемикачі керуються бітами "тегу" порту призначення
• Обмеження: складність реалізації для великої кількості портів
• Якщо порт зайнятий → буферизація + повторна спроба
```

#### 2️⃣ Загальна шина (Shared Bus)

```
[Порт 1] ↔ [Високошвидкісна шина] ↔ [Порт N]
                ↑
        Розподіл часу (TDM)

• Усі порти підключені до спільної шини
• Вимога: пропускна здатність шини ≥ Σ пропускних здатностей портів
• Кадр передається частинами (осередками) → псевдопаралельність
• Ризик: шина може стати "вузьким місцем" при високому навантаженні
```

#### 3️⃣ Пам'ять, що розділяється (Shared Memory)

```
[Вхідні блоки портів] → [Спільна пам'ять] → [Вихідні блоки портів]
                              ↑
                    [Менеджер черг]

• Дані зберігаються в єдиній буферній пам'яті
• Менеджер організує окремі черги для кожного вихідного порту
• Перевага: гнучке розподілення буфера між портами
• Вимога: висока швидкість пам'яті для підтримки wire-speed
```

### 🔹 Порівняння архітектур

| Архітектура | Швидкість | Масштабованість | Вартість | Застосування |
|-------------|-----------|-----------------|----------|--------------|
| **Матриця** | ⭐⭐⭐⭐⭐ | Обмежена (до ~48 портів) | Висока | Ядро мережі, високе навантаження |
| **Шина** | ⭐⭐⭐ | Середня | Середня | Середній рівень, модульні комутатори |
| **Пам'ять** | ⭐⭐⭐⭐ | Висока (залежить від пам'яті) | Середня/Висока | Універсальне, enterprise-рівень |

## 6. Інтелектуальні комутатори та VLAN

### 🔹 Комутація 3-го рівня: два підходи

```
┌─────────────────────────────────────────┐
│ 🔹 Routing Switches (маршрутизуючі комутатори) │
├─────────────────────────────────────────┤
• Основна робота: 2-й рівень (апаратно)
• Пошук маршруту: 3-й рівень (кешування, зовнішній маршрутизатор)
• НЕ використовують RIP/OSPF/BGP напряму
• Протокольно-незалежні
• Швидкість: близька до wire-speed

┌─────────────────────────────────────────┐
│ 🔹 Switching Routers (комутуючі маршрутизатори) │
├─────────────────────────────────────────┤
• Control-plane: процесор загального призначення (RIP/OSPF/BGP)
• Data-plane: ASIC-акселератори (апаратна комутація)
• Повна підтримка протоколів маршрутизації
• Продуктивність: wire-speed для відомих протоколів
```

### 🔹 Віртуальні мережі (VLAN — Virtual LAN)

```
📌 Визначення:
VLAN — логічна група вузлів, що утворюють окремий 
домен широкомовного трафіку (Broadcast Domain) 
на канальному рівні.

🎯 Переваги:
✅ Локалізація широкомовного трафіку (захист від "broadcast storm")
✅ Підвищення безпеки (ізоляція на рівні L2)
✅ Гнучке групування за функціями, а не фізичним розташуванням
✅ Спрощення управління змінами (переміщення користувача → зміна VLAN на порту)
```

### 🔹 Способи реалізації VLAN

#### 🔸 Port-based VLAN (угруповання портів)

```
Принцип: кожен фізичний порт комутатора належить до одного (або кількох) VLAN

Приклад конфігурації:
┌─────────────────┐
│ Комутатор       │
├─────────────────┤
│ Порти 1-8  → VLAN 10 (Відділ продажів)    │
│ Порти 9-16 → VLAN 20 (Бухгалтерія)        │
│ Порти 17-24 → VLAN 30 (IT-відділ)         │
└─────────────────┘

✅ Переваги:
• Проста конфігурація (через CLI/Web-інтерфейс)
• Не вимагає підтримки 802.1Q на кінцевих пристроях
• Можливість спільних ресурсів (порт у кількох VLAN)

❌ Обмеження:
• Погана масштабованість (працює в межах одного комутатора/стека)
• Не підходить для розподілених мереж з багатьма комутаторами
```

#### 🔸 IEEE 802.1Q (Tagged VLAN) — стандартний підхід

```
🔧 Зміна формату кадру (+4 байти):

┌────────┬────────┬──────┬────────┬────────┬────────┐
| DA (6) | SA (6) | TPID |  TCI   | Type   | Data...|
└────────┴────────┴──────┴────────┴────────┴────────┘
                     ↑      ↑
                  0x8100  ┌─ Priority (3 біти) — 8 рівнів (0-7)
                          ├─ CFI (1 біт) — Canon. Format (0 для Ethernet)
                          └─ VID (12 біт) — VLAN ID (1–4094)

📊 VLAN ID:
• Діапазон: 0–4095
• Доступні: 1–4094 (4094 VLAN)
• Зарезервовані: 0 (пріоритет), 4095 (службовий)

🔄 Обробка кадрів на портах:

Тип порту      | Вхід (від ПК)    | Вихід (до ПК)    | Внутрішня обробка
---------------|------------------|------------------|------------------
Access (Untagged) | Приймає Untagged | Надсилає Untagged | Додає/видаляє тег
Trunk (Tagged)    | Приймає Tagged   | Надсилає Tagged   | Зберігає тег
Hybrid            | Обидва типи      | Обидва типи      | Гнучка політика
```

> 💡 **Практична порада**: 
> - Використовуйте **Access-порти** для кінцевих пристроїв (ПК, принтери)
> - Використовуйте **Trunk-порти** для з'єднання комутаторів між собою
> - Завжди явно вказуйте **Native VLAN** на Trunk-портах (за замовчуванням = VLAN 1)

## 7. Класифікація ЛОМ за IEEE 802

### 🔹 Структура стандартів IEEE 802

```
IEEE 802 — сімейство стандартів для локальних та міських мереж
Канальний рівень розділено на два підрівні:

┌─────────────────────────────┐
│   LLC (Logical Link Control)│ ← IEEE 802.2
│   • Інтерфейс до мережевого │
│   • Сервіси: без встановлення / з встановленням з'єднання
├─────────────────────────────┤
│   MAC (Media Access Control)│ ← Залежить від технології
│   • Доступ до середовища    │
│   • Адресація, форматування кадрів
└─────────────────────────────┘
```

### 🔹 Огляд ключових стандартів

| Стандарт | Назва | Опис |
|----------|-------|------|
| **802.1** | Higher Layer LAN Protocols | Управління мережею, міжмережева взаємодія |
| → 802.1D | Spanning Tree Protocol (STP) | Усунення петель у комутаторах |
| → 802.1Q | VLAN Tagging | Мітка кадру для віртуальних мереж |
| → 802.1p | Prіority Tagging | Пріоритезація трафіку (3 біти в TCI) |
| **802.2** | LLC | Уніфікований інтерфейс для всіх 802.* технологій |
| **802.3** | Ethernet (CSMA/CD) | Класичний Ethernet, Fast/Gigabit Ethernet |
| **802.4** | Token Bus | Мережі з маркерним доступом на шині (застарілий) |
| **802.5** | Token Ring | Кільцева мережа з маркером (IBM, застарілий) |
| **802.11** | Wireless LAN | Wi-Fi (a/b/g/n/ac/ax) |
| **802.15** | WPAN | Bluetooth, ZigBee (персональні мережі) |
| **802.16** | WiMAX | Широкосмуговий бездротовий доступ |

### 🔹 Сервіси LLC (802.2)

```
┌─────────────────────────────────────────┐
│ 🔹 Тип 1 (LLC1): Без встановлення, без підтверджень │
├─────────────────────────────────────────┤
• Найпростіший, ненадійний
• Використовується з надійними протоколами вищих рівнів (TCP/IP)
• Кадр: UI (Unnumbered Information)

┌─────────────────────────────────────────┐
│ 🔹 Тип 2 (LLC2): З встановленням з'єднання + підтвердження │
├─────────────────────────────────────────┤
• Надійний, з контролем потоку та помилок
• Використовувався в NetWare, SNA
• Кадри: SABME (запит), UA (підтвердження), I (інформаційні), RR/REJ (керування)

┌─────────────────────────────────────────┐
│ 🔹 Тип 3 (LLC3): Без встановлення, з підтвердженням │
├─────────────────────────────────────────┤
• Компроміс: швидше за Тип 2, надійніше за Тип 1
• Кадр: UI + підтвердження прийому
```

## 🔑 Ключові висновки

1. **100Base-T4** — "міст" між старим (Cat.3) і новим (Fast Ethernet), але обмежений напівдуплексом.
2. **Обмеження топології** у Fast Ethernet (~205 м) компенсується використанням **комутаторів** та **повнодуплексної оптики**.
3. **MAC-адреса** має вбудовану логіку (біти I/G, U/L) для підтримки унікальності, групування та локального керування.
4. **Комутатори** поділяють домени колізій, що різко підвищує продуктивність порівняно з концентраторами.
5. **VLAN (802.1Q)** — стандартний спосіб логічної ізоляції на L2; використовує 4-байтову мітку з 12-бітовим VID (4094 VLAN).
6. **L3-комутація** поєднує швидкість L2 з функціональністю L3: routing switches (кешування) та switching routers (ASIC + CPU).
7. **IEEE 802.2 (LLC)** забезпечує уніфікований інтерфейс, але в сучасних мережах часто "прозорий" завдяки домінуванню Ethernet II + IP.

## 🛠️ Практичні команди (приклад для Cisco)

```bash
# Налаштування VLAN на комутаторі
Switch> enable
Switch# configure terminal

# Створення VLAN
Switch(config)# vlan 10
Switch(config-vlan)# name Sales
Switch(config-vlan)# exit

# Призначення порту до VLAN (Access)
Switch(config)# interface gigabitEthernet 0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 10
Switch(config-if)# exit

# Налаштування Trunk-порту (802.1Q)
Switch(config)# interface gigabitEthernet 0/24
Switch(config-if)# switchport mode trunk
Switch(config-if)# switchport trunk native vlan 99
Switch(config-if)# switchport trunk allowed vlan 10,20,30
Switch(config-if)# exit

# Перевірка конфігурації
Switch# show vlan brief
Switch# show interfaces trunk
```
